

1. **Скачивает VPN-конфиги** из открытых источников, перечисленных в `sources.txt` (поддерживаются тысячи источников на GitHub и других платформах). Поддерживаются протоколы VLESS, VMess, Trojan, Shadowsocks, Hysteria2.

2. **Очищает и валидирует** найденные конфиги:
   - Проверяет корректность UUID, структуры URL и обязательных параметров.
   - Удаляет дубликаты (по содержимому URL без учёта имени конфига).

3. **Фильтрует по SNI и домену**: оставляет только конфиги, чей SNI/Host совпадает с доменами российских CDN и сервисов из `domain.txt` (или `whitelist.txt`, если он есть). Это гарантирует, что конфиги работают при использовании белых списков.

4. **Переименовывает конфиги** для удобства: в имя включается протокол (WS, gRPC, TCP, VMess, Trojan, SS, HY2 и т.д.) и название сервиса (например, «VK», «Яндекс», «Wildberries»). Добавляется метка `[#РКП]`.

5. **Кодирует специальные символы** в URL для совместимости с разными VPN-клиентами.

6. **Проверяет работоспособность** каждого конфига через ядро [Xray-core](https://github.com/XTLS/Xray-core):
   - Xray скачивается автоматически при первом запуске (Windows, Linux, macOS).
   - Для каждого конфига поднимается временный SOCKS5-прокси и делается тестовый HTTP-запрос.
   - Для Hysteria2 (не поддерживается Xray) используется TCP + TLS-рукопожатие.
   - Нерабочие конфиги исключаются из итогового списка.

7. **Чёрный список (notworkers)**: нерабочие конфиги сохраняются в SQLite-базу `configs/notworkers.db` с TTL-таймером (7 дней) — при следующих запусках они пропускаются, экономя время. Ожившие конфиги автоматически удаляются из чёрного списка. Конфиги с серверами не в России попадают в чёрный список **навсегда** — они не удаляются по TTL и не исключаются при «оживании», так как серверы за пределами России бесполезны для обхода белых списков.

8. **Сохраняет рабочие конфиги** в файл `configs/url_work.txt`, отсортированные по пингу (от быстрых к медленным).

### 🔍 Система фильтрации конфигов

Парсер использует многоступенчатую систему фильтрации, чтобы из тысяч найденных конфигов оставить только те, которые реально работают при наличии белых списков в России:

```
Источники (sources.txt)
    │
    ▼
① Очистка и валидация
    │  Проверка UUID, структуры URL, удаление дубликатов
    ▼
② Фильтр по SNI/домену (domain.txt)
    │  Оставляет только конфиги с SNI российских CDN и сервисов
    │  (VK, Яндекс, Wildberries, Rutube и др.)
    ▼
③ Чёрный список (notworkers.db)
    │  Пропускает конфиги, которые ранее не работали
    │  (SQLite, TTL 7 дней, минимум 2 провала подряд)
    │  GEO-заблокированные серверы хранятся навсегда
    ▼
④ GEO-предфильтр (ip-api.com)
    │  Определяет страну сервера по IP
    │  Оставляет только Russia и Unknown
    │  Остальные (US, DE, NL...) → в чёрный список НАВСЕГДА
    ▼
⑤ TCP/TLS предфильтр
    │  Проверяет доступность host:port (200 потоков)
    │  Мёртвые хосты → в чёрный список
    ▼
⑥ Xray-проверка (полная)
    │  Поднимает SOCKS5-прокси для каждого конфига
    │  Делает тестовый HTTP-запрос через прокси
    │  Строгий режим: 2 из 3 запросов должны пройти
    │  Проверка стабильности: 2 дополнительных запроса
    ▼
⑦ Сортировка по скорости
    │  Рабочие конфиги сортируются по пингу
    ▼
configs/url_work.txt — итоговый список
configs/url_work_speed.txt — топ-30 быстрых
```

Каждый этап значительно сокращает список:
- **Этап ②** убирает ~90% конфигов (несовместимые SNI)
- **Этап ③** пропускает ранее проверенные нерабочие (~10-30%)
- **Этап ④** отсеивает серверы не в России (~40-70%)
- **Этап ⑤** убирает мёртвые хосты (~10-20%)
- **Этап ⑥** оставляет только реально работающие конфиги

### Поддерживаемые протоколы

| Протокол | Транспорт | Метод проверки |
|---|---|---|
| **VLESS + Reality** | TCP | Xray-core |
| **VLESS + TLS** | TCP/WS/gRPC/XHTTP/H2 | Xray-core |
| **VMess + TLS** | TCP/WS/gRPC | Xray-core |
| **Trojan** | TCP/WS/gRPC | Xray-core |
| **Shadowsocks** | TCP | Xray-core |
| **Hysteria2** | QUIC | TCP + TLS-рукопожатие |

### Используемые файлы

| Файл | Назначение |
|---|---|
| `sources.txt` | Список URL-источников с VPN-конфигами |
| `domain.txt` | Список доменов для фильтрации по SNI |
| `iP.txt` | Список IP-адресов российских CDN |
| `whitelist.txt` | (опционально) Пользовательский список доменов для фильтрации |
| `configs/url_work.txt` | Итоговый список рабочих конфигов (обновляется автоматически) |
| `configs/url_work_speed.txt` | Топ-30 самых быстрых конфигов |
| `configs/notworkers.db` | Чёрный список нерабочих конфигов (SQLite, TTL 7 дней) |

## 🚀 Как запустить парсер

### Требования
- Windows, Linux или macOS — Xray загружается автоматически для текущей ОС
- Python 3.9 или новее
- Интернет-соединение

### Установка зависимостей
```bash
pip install -r requirements.txt
```

### Запуск

**Непрерывный режим** (запускает циклы каждый час):
```bash
python parser.py
```

**Однократный запуск** (один цикл и завершение):
```bash
python parser.py --once
```

### Настройки

В файле `config.ini` можно изменить параметры:

| Параметр | По умолчанию | Описание |
|---|---|---|
| `threads_download` | `50` | Параллельных потоков для скачивания |
| `xray_max_workers` | `30` | Параллельных потоков для Xray-тестирования |
| `xray_timeout` | `3` | Таймаут теста конфига (секунды) |
| `strong_style_attempts` | `3` | Для VLESS: сколько подряд успешных запросов к `generate_204` требуется |
| `strong_style_timeout` | `12` | Для VLESS: таймаут одного запроса в строгом режиме (секунды) |
| `strong_max_response_time` | `3` | Для VLESS: максимальное допустимое время ответа (секунды) |
| `notworkers_enabled` | `true` | Включить/отключить чёрный список |
| `notworkers_ttl_days` | `7` | Срок хранения записи в чёрном списке (дни) |

## 🤖 Автоматическое обновление на GitHub

Репозиторий настроен на **автоматический запуск парсера через GitHub Actions** каждые ~25 минут.

Рабочий процесс (`.github/workflows/update_configs.yml`):
1. Запускается на Windows-раннере GitHub.
2. Устанавливает Python и зависимости.
3. Выполняет парсер в режиме `--once`.
4. Копирует результат в `configs/url_work.txt` и `configs/url_work_speed.txt`.
5. Коммитит обновление `configs/notworkers.db`.
6. Создаёт git-коммит и пушит обновление в репозиторий.

Вы также можете запустить обновление вручную через вкладку **Actions** → **Обновление конфигов VPN** → **Run workflow**.
